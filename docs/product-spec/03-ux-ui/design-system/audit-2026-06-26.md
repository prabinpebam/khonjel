# Design System & Library — Audit (2026-06-26)

> **Status:** findings + opportunities, pre-implementation. Measures the live code in
> `app/src` against the design-system contract in [`01-intent.md`](01-intent.md) (P1–P13) and
> the `ds-lint` rules in [`scripts/ds-lint.mjs`](../../../../app/scripts/ds-lint.mjs).
> **Method:** static survey of `styles/`, `components/ui`, `components/common`, `surfaces/`,
> and `features/` (token usage, inline styles, arbitrary utilities, component reuse vs.
> reinvention, layering). No code was changed.

---

## 1. Verdict

The **foundations are strong**: a clean three-tier token layer, a working `ds-lint` gate,
CVA-driven `Button`, theme via `[data-theme]` token overrides (no component branches on
theme), logical RTL-safe props, and data registries in `config/`. The system is well
*specified*.

The **gap is enforcement of reuse**. The contract leans entirely on **Storybook as the
inventory** (P3/P10/P13) — and **Storybook does not exist**. With no discoverable inventory,
several shared primitives have gone **dead while features reinvent them inline**, and a few
**compound patterns are now duplicated across 3–5 screens**. None of this is catastrophic,
but it is exactly the drift P1–P13 were written to prevent, and it compounds every session.

Severity legend: **P0** fix now (cheap, stops fresh drift) · **P1** high-value reuse work ·
**P2** structural/process.

| # | Finding | Principle | Severity |
|---|---|---|---|
| 1 | No Storybook / inventory exists | P3, P10, P13 | P2 |
| 2 | Shared primitives dead, reinvented inline (`Progress`, `EmptyState`, `Placeholder`) | P3, P11 | P1 |
| 3 | `ThemeToggle` reinvents the existing `Segmented` control | P5, P11 | **P0** |
| 4 | `Card` used inconsistently; card cluster hand-rolled | P11 | P1 |
| 5 | Repeated compound patterns never extracted (search field, panel, row-actions, inline-rename) | P11 | P1 |
| 6 | `ds-lint` blind spots (non-px arbitrary values, reuse, layering) | P1, P4, P11 | **P0** |
| 7 | Instance tokens not consistently consumed (`--rail-width`) | P1a | P2 |
| 8 | i18n contract unmet (hardcoded copy) | P8 | P2 |
| 9 | Large feature files embed reusable sub-components | P4, P13 | P2 |

## 2. What's working (keep)

- **Token layer (P1/P1a).** [`styles/tokens.css`](../../../../app/src/styles/tokens.css) is the
  single home for raw color/radius/shadow/motion; [`styles/globals.css`](../../../../app/src/styles/globals.css)
  maps them via `@theme inline` so `[data-theme]` re-themes everything with zero component
  edits. Instance dims (`--sidebar-width`, `--titlebar-height`, `--content-max-width`) are
  correctly separated from the scale.
- **`ds-lint` (L1).** Catches hex/rgb literals, bare-px utilities, `!important`, magic
  z-index, and emoji. Real, enforced discipline most teams lack.
- **Variants as data (P5).** `Button` is CVA ([`button-variants.ts`](../../../../app/src/components/ui/button-variants.ts)).
- **Theme via tokens (P6).** One `useThemeStore`; `ThemeProvider` flips the root attribute;
  no `theme === 'dark' ? …` anywhere.
- **Logical props (P8 direction).** `ps/pe/ms/me/start/end` used throughout.
- **Data in `config/` (P12).** `nav.ts`, `settings.ts` are registries, not JSX literals.
- **Accessible custom `Select`.** Portaled, keyboard-driven, now `role="listbox/option"`.

## 3. Findings

### 3.1 — No Storybook / inventory (P3, P10, P13) — P2
The contract says it five times: *"Storybook is the authoritative inventory… Not in
Storybook = not real… Agents discover what's available by reading the inventory, not by
grepping screens."* There are **zero `*.stories.*` files**. Consequence: there is no way —
for a human or an agent — to see what already exists, which is the **root cause of findings
3.2–3.5**. Nothing fails when a shared component is added without a story, and nothing
surfaces a dead component.

**Recommend:** stand up Storybook (or, lighter, an in-app `?surface=inventory` "kitchen
sink" route reusing the real components) with one story per `ui/` + `common/` member, and a
CI check that every shared component is referenced by a story.

### 3.2 — Dead shared primitives, reinvented inline (P3, P11) — P1
Components exist in the library but have **no importers**, while the same UI is hand-rolled
in features/surfaces:

- **`Progress`** ([`ui/progress.tsx`](../../../../app/src/components/ui/progress.tsx)) — unused.
  Progress bars are re-implemented with inline `style={{ width }}` in
  [`Insights.tsx`](../../../../app/src/features/insights/Insights.tsx#L94),
  [`LocalModelList.tsx`](../../../../app/src/surfaces/settings/LocalModelList.tsx#L204),
  [`LocalModelSetup.tsx`](../../../../app/src/surfaces/settings/LocalModelSetup.tsx#L282),
  and [`AccelerationCard.tsx`](../../../../app/src/surfaces/settings/AccelerationCard.tsx#L172).
- **`EmptyState`** ([`common/EmptyState.tsx`](../../../../app/src/components/common/EmptyState.tsx)) —
  unused. Empty states are inlined ad-hoc (Notes "No notes in this folder.", Chat "Ask me
  anything." / "No conversations match.", etc.).
- **`Placeholder`** ([`common/Placeholder.tsx`](../../../../app/src/components/common/Placeholder.tsx)) —
  appears unused now that all primary views are built.

**Recommend:** adopt `Progress` + `EmptyState` at the call sites; delete `Placeholder` (or
fold it into `EmptyState` as a no-icon variant).

### 3.3 — `ThemeToggle` reinvents `Segmented` (P5, P11) — P0 (self-inflicted)
[`ui/segmented.tsx`](../../../../app/src/components/ui/segmented.tsx) is the canonical
segmented control — its doc comment literally reads *"e.g. Theme: Light / Dark / Auto"* — and
it is already used at [`GeneralSettings.tsx#L67`](../../../../app/src/surfaces/settings/sections/GeneralSettings.tsx#L67)
(`<Segmented<Theme>>`). The new [`common/ThemeToggle.tsx`](../../../../app/src/components/common/ThemeToggle.tsx)
(added this session for the title bar) **hand-rolls a second segmented control**. Two
implementations for one role — the exact P5 anti-pattern. (The old MockStudio pill had the
same flaw; extracting it perpetuated it.)

**Recommend:** add a compact / icon-only `size` variant to `Segmented`, then make
`ThemeToggle` a thin composition of `<Segmented>`. One control, shared by the title bar and
Settings ▸ Appearance.

### 3.4 — `Card` used inconsistently (P11) — P1
`<Card>` ([`ui/card.tsx`](../../../../app/src/components/ui/card.tsx)) is used in Insights,
Dictionary, Transforms, and `StatCard` — but the **same cluster** `rounded-md border
border-border bg-surface p-4/5` (frequently **missing `shadow-card`**, so it doesn't even
match `Card`) is hand-rolled in
[`Upload.tsx#L156`](../../../../app/src/features/upload/Upload.tsx#L156),
[`LocalModelSetup.tsx#L212`](../../../../app/src/surfaces/settings/LocalModelSetup.tsx#L212),
[`AccelerationCard.tsx#L62`](../../../../app/src/surfaces/settings/AccelerationCard.tsx#L62),
and [`ConnectionsSettings.tsx#L204`](../../../../app/src/surfaces/settings/sections/ConnectionsSettings.tsx#L204).

**Recommend:** replace hand-rolled clusters with `<Card>`; if a borderless/elevation-less
inset is wanted, add a `variant` to `Card` rather than re-typing the classes.

### 3.5 — Compound patterns duplicated, never promoted (P11) — P1
P11 says *"promote to common/ when a **second** feature needs it."* These already have 2–5
consumers and remain copy-pasted class clusters:

- **(a) Search field** (leading `Search` icon + clearable `X`): identical icon cluster in
  [`Chat.tsx#L272`](../../../../app/src/features/chat/Chat.tsx#L272),
  [`Notes.tsx#L174`](../../../../app/src/features/notes/Notes.tsx#L174),
  [`Dictionary.tsx#L216`](../../../../app/src/features/dictionary/Dictionary.tsx#L216).
  → **`SearchInput`** (`value`, `onChange`, `onClear`).
- **(b) Full-height bordered panel** (`rounded-lg border border-border bg-surface
  overflow-hidden flex flex-col`): [`Chat.tsx#L265`](../../../../app/src/features/chat/Chat.tsx#L265)/L378,
  [`Notes.tsx#L171`](../../../../app/src/features/notes/Notes.tsx#L171)/L276/L306. →
  **`Panel`** with header/body slots (distinct from `Card`: full-height, no shadow, clips).
- **(c) Row hover-actions** (`opacity-0 transition-opacity group-hover:opacity-100`): Home,
  Chat (×3), Notes, Dictionary, Transforms — 7+ sites. → a `RowActions` wrapper or a single
  documented utility class.
- **(d) Inline-rename autofocus input** (`ref={(el) => el?.focus()}`, plus select-on-focus):
  [`Chat.tsx#L325`](../../../../app/src/features/chat/Chat.tsx#L325)/L532,
  [`Notes.tsx#L223`](../../../../app/src/features/notes/Notes.tsx#L223). → `useAutoFocus()`
  hook (or an `InlineEdit` control).

### 3.6 — `ds-lint` blind spots (P1, P4, P11) — P0
The gate is good but narrow. It bans bare-`px` arbitrary utilities yet **not** other units,
so scale-bypassing values slip through:
- `max-w-[72%]` / `w-[72%]` — [`Chat.tsx#L568`](../../../../app/src/features/chat/Chat.tsx#L568)/L538 (a magic bubble width, repeated).
- `text-[0.85em]` — [`Markdown.tsx#L20`](../../../../app/src/features/chat/Markdown.tsx#L20) (bypasses the type scale).
- `min-h-[60vh]` — [`Placeholder.tsx#L9`](../../../../app/src/components/common/Placeholder.tsx#L9).

It also can't see the **structural** problems above (reinvented primitives, dead components,
cross-feature imports).

**Recommend:**
1. Widen the `arbitrary-px` rule to **all** length units (`px|rem|em|vh|vw|ch|%`), allowing
   only `-[var(--token)]` and fraction utilities (`1/2`). Add a token for the chat-bubble
   max width instead of `72%`.
2. Add a cheap heuristic rule flagging the known clusters (card/panel/search-icon) so
   reinvention is caught at L1.
3. Add an ESLint `no-restricted-imports` rule: `features/*` may not import another
   `features/*` (enforce P4 layering).

### 3.7 — Instance tokens not consumed (P1a) — P2
`--rail-width: 260px` exists but Chat's rail is `w-60` (240px) and panel widths are ad-hoc
(`w-52`, `w-60`, `w-72`). Either consume the token (`w-[var(--rail-width)]`) or delete it so
it isn't a stale "closest token" trap.

### 3.8 — i18n contract unmet (P8) — P2
P8 forbids hardcoded user-facing strings in shared components (react-i18next). All copy is
hardcoded; there is no i18n layer. This is likely a deliberate scope call — but it is a
standing deviation. **Decide:** adopt i18n, or amend P8 to "English-only for now" so the
contract matches reality.

### 3.9 — Feature files embed reusable sub-components (P4, P13) — P2
[`Chat.tsx`](../../../../app/src/features/chat/Chat.tsx) (~600 lines) defines `MessageBubble`
and `HighlightText` inline; there are no `features/{id}/components/` folders. Not a hard
violation (single-feature use), but `HighlightText` is a generic match-highlighter and
`MessageBubble` is testable in isolation — extracting them aids discoverability and unit
testing.

## 4. Recommended sequence

**P0 — stop fresh drift (small, do first)**
1. `Segmented` gains a compact/icon-only `size`; `ThemeToggle` composes it (3.3).
2. Harden `ds-lint`: all-unit arbitrary-value ban + the cross-feature import ESLint rule
   (3.6). Add a `--chat-bubble-max` token; fix `text-[0.85em]`.

**P1 — collapse duplication via reuse**
3. Extract `SearchInput` and `Panel`; adopt them in Chat/Notes/Dictionary (3.5a/b).
4. Adopt `Progress` and `EmptyState` at all inline sites; delete `Placeholder` (3.2).
5. Replace hand-rolled card clusters with `<Card>` (+ a `variant` if needed) (3.4).
6. Extract `useAutoFocus` + a `RowActions` helper (3.5c/d).

**P2 — close the inventory + process gap**
7. Stand up Storybook (or an inventory route) with a story per shared component, and a CI
   "every shared component has a story" check (3.1).
8. Decide the i18n posture (3.8); extract `HighlightText`/`MessageBubble` (3.9); reconcile
   `--rail-width` (3.7).

Each P0/P1 item is independently shippable behind `verify:quick` + the EDD suite.
